## Описание проекта

В рамках практического занятия №12 на базе сервисов **Tasks** (REST) и **GraphQL** из предыдущей работы проводится сравнение двух подходов к разработке API.

**Цель работы:** Понять сильные и слабые стороны REST и GraphQL на практике: как меняется клиентский сценарий, сколько запросов нужно, как устроены ошибки, что легче документировать и кэшировать.

**Объекты сравнения:**
- **REST API** — сервис `tasks` на порту 8082
- **GraphQL API** — сервис `graphql` на порту 8090

## Выбор сценария для сравнения

Для сравнения выбран **реалистичный UI-сценарий "Экран списка + карточка детали"**:

| Экран | Требуемые поля |
|-------|----------------|
| Список задач | `id`, `title`, `done` |
| Детали задачи | `id`, `title`, `description`, `done` |
| Действие | Создать задачу с `title` и `description` |

Этот сценарий приближен к реальному приложению и позволяет наглядно продемонстрировать различия в количестве запросов и объёме передаваемых данных.

## Реализация сценария через REST

### 1. Получение списка задач (для экрана списка)

**Запрос:**
```
GET http://localhost:8082/v1/tasks
Headers:
  Authorization: Bearer demo-token
  Cookie: session_id=demo-session-123
```

**Ожидаемый ответ:**
```json
[
  {
    "id": "t_1",
    "title": "Task 1",
    "description": "Description 1",
    "due_date": "",
    "done": false
  },
  {
    "id": "t_2",
    "title": "Task 2",
    "description": "Description 2",
    "due_date": "",
    "done": true
  }
]
```

**Проблема:** Поле `description` не требуется на экране списка, но сервер его возвращает (over-fetching).

### 2. Получение деталей задачи (по ID)

**Запрос:**
```
GET http://localhost:8082/v1/tasks/t_1
Headers:
  Authorization: Bearer demo-token
  Cookie: session_id=demo-session-123
```

**Ожидаемый ответ:**
```json
{
  "id": "t_1",
  "title": "Task 1",
  "description": "Description 1",
  "due_date": "",
  "done": false
}
```

### 3. Создание новой задачи

**Запрос:**
```
POST http://localhost:8082/v1/tasks
Headers:
  Content-Type: application/json
  Authorization: Bearer demo-token
  Cookie: session_id=demo-session-123
  X-CSRF-Token: demo-csrf-456

Body:
{
  "title": "REST Task",
  "description": "Created via REST"
}
```

**Ожидаемый ответ (201 Created):**
```json
{
  "id": "t_3",
  "title": "REST Task",
  "description": "Created via REST",
  "due_date": "",
  "done": false
}
```

### Итог по REST для сценария

| Действие | Количество запросов | Объём данных (примерно) |
|----------|---------------------|-------------------------|
| Показать список задач | 1 запрос | ~150-200 байт на задачу (с лишними полями) |
| Показать детали задачи | 1 запрос | ~100-150 байт |
| Создать задачу | 1 запрос | ~100 байт |
| **Всего для сценария** | **3 запроса** | **~350-450 байт** |

## Реализация сценария через GraphQL

### 1. Получение списка задач (только нужные поля)

**Запрос:**
```
POST http://localhost:8090/query
Headers:
  Content-Type: application/json

Body:
{
  "query": "query { tasks { id title done } }"
}
```

**Ожидаемый ответ:**
```json
{
  "data": {
    "tasks": [
      {
        "id": "t_1",
        "title": "Task 1",
        "done": false
      },
      {
        "id": "t_2",
        "title": "Task 2",
        "done": true
      }
    ]
  }
}
```

### 2. Получение деталей задачи (с выбором полей)

**Запрос:**
```
POST http://localhost:8090/query
Headers:
  Content-Type: application/json

Body:
{
  "query": "query($id: ID!) { task(id: $id) { id title description done } }",
  "variables": {
    "id": "t_1"
  }
}
```

**Ожидаемый ответ:**
```json
{
  "data": {
    "task": {
      "id": "t_1",
      "title": "Task 1",
      "description": "Description 1",
      "done": false
    }
  }
}
```

### 3. Создание новой задачи

**Запрос:**
```
POST http://localhost:8090/query
Headers:
  Content-Type: application/json

Body:
{
  "query": "mutation($input: CreateTaskInput!) { createTask(input: $input) { id title description done } }",
  "variables": {
    "input": {
      "title": "GraphQL Task",
      "description": "Created via GraphQL"
    }
  }
}
```

**Ожидаемый ответ:**
```json
{
  "data": {
    "createTask": {
      "id": "t_3",
      "title": "GraphQL Task",
      "description": "Created via GraphQL",
      "done": false
    }
  }
}
```

### Итог по GraphQL для сценария

| Действие | Количество запросов | Объём данных (примерно) |
|----------|---------------------|-------------------------|
| Показать список задач | 1 запрос | ~100-120 байт на задачу (только нужные поля) |
| Показать детали задачи | 1 запрос | ~100-120 байт |
| Создать задачу | 1 запрос | ~100-120 байт |
| **Всего для сценария** | **3 запроса** | **~300-360 байт** |

## Сравнительный анализ

### Таблица сравнения

| Критерий | REST | GraphQL |
|----------|------|---------|
| **Количество запросов** | 3 запроса (GET /tasks, GET /tasks/{id}, POST /tasks) | 3 запроса (query tasks, query task, mutation createTask) |
| **Over-fetching** | Присутствует — в списке возвращаются лишние поля (description) | Отсутствует — клиент запрашивает только нужные поля |
| **Under-fetching** | Отсутствует — все данные для экрана получаются за 1 запрос | Отсутствует — все данные для экрана получаются за 1 запрос |
| **Объём данных** | Больше из-за лишних полей | Минимальный, только запрошенные поля |
| **Типизация** | Неявная, описывается в документации | Явная через схему GraphQL |
| **Документирование** | Требуется внешняя документация (Swagger) | Схема является самодокументированной (introspection) |
| **Обработка ошибок** | HTTP статусы (200, 201, 400, 404, 500) + тело ошибки | HTTP 200 всегда, ошибки в поле `errors` |
| **Кэширование** | Простое — по URL (HTTP кэширование) | Сложное — единая точка входа, требуется кэширование на уровне данных |
| **Валидация запросов** | На стороне сервера | На стороне сервера + валидация схемы |
| **Версионирование** | Через URL (/v1/, /v2/) | Эволюция схемы с обратной совместимостью |

### Анализ по пунктам

#### 5.1. Количество запросов
Оба подхода требуют **одинаковое количество запросов** (3) для выполнения полного сценария. Однако в более сложных сценариях с вложенными данными GraphQL может сократить количество запросов (например, запрос задачи вместе с комментариями одним запросом против нескольких REST запросов).

#### 5.2. Объём данных
**REST:** Возвращает все поля, даже те, которые не нужны клиенту. В нашем примере в списке задач возвращается поле `description`, которое не используется на экране списка.

**GraphQL:** Возвращает только запрошенные поля. В списке задач возвращаются только `id`, `title`, `done`. Экономия трафика особенно важна для мобильных устройств.

#### 5.3. Ошибки и статусы

**REST:**
- Ошибки выражаются HTTP статусами:
    - `200 OK` — успех
    - `201 Created` — создано
    - `400 Bad Request` — неверный запрос
    - `401 Unauthorized` — не авторизован
    - `404 Not Found` — не найдено
    - `500 Internal Server Error` — ошибка сервера
- Тело ответа содержит детали ошибки в JSON

**GraphQL:**
- Все запросы возвращают HTTP `200 OK`
- Ошибки передаются в поле `errors` ответа:
```json
{
  "errors": [
    {
      "message": "task not found",
      "path": ["task"],
      "extensions": {
        "code": "NOT_FOUND"
      }
    }
  ],
  "data": {
    "task": null
  }
}
```

**Сравнение удобства:**
- REST: проще мониторить по HTTP статусам, но клиенту нужно парсить разные статусы
- GraphQL: единый формат ответа, но сложнее мониторинг (все запросы 200)

#### 5.4. Кэширование (концептуально)

**REST:**
- Простое кэширование на уровне HTTP
- Разные URL для разных ресурсов
- Можно использовать заголовки `Cache-Control`, `ETag`
- CDN могут кэшировать ответы

**GraphQL:**
- Единая точка входа (/query)
- Кэширование на уровне HTTP неэффективно
- Требуется кэширование на уровне данных (например, Redis)
- Persisted queries могут помочь с кэшированием

## Выводы

В ходе выполнения практического занятия №12 было проведено сравнение REST и GraphQL на примере одного и того же функционала управления задачами.

### Основные результаты

1. **Количество запросов** — для простого сценария оба подхода требуют одинаковое количество запросов (3). Однако при работе со сложными связанными данными GraphQL может сократить количество запросов (under-fetching).

2. **Объём данных** — GraphQL показал экономию трафика за счёт отсутствия over-fetching. В нашем примере экономия составила около 33% на запросе списка задач. Это особенно важно для мобильных приложений с ограниченным трафиком.

3. **Типизация и документация** — GraphQL предоставляет строгую типизацию через схему, которая является самодокументированной. REST требует внешних инструментов (Swagger/OpenAPI).

4. **Обработка ошибок** — REST использует HTTP статусы, что удобно для мониторинга, но требует обработки разных кодов. GraphQL унифицирует ответы (всегда 200), но усложняет мониторинг.

5. **Кэширование** — REST имеет преимущество благодаря встроенному HTTP кэшированию. GraphQL требует реализации кэширования на уровне приложения.

### Когда выбирать REST

- Простые CRUD приложения
- Требуется эффективное HTTP кэширование
- Нужна простая интеграция с CDN
- API в основном используется для мобильных приложений с простыми запросами
- Команда уже имеет опыт работы с REST

### Когда выбирать GraphQL

- Сложные клиенты с разными потребностями в данных
- Мобильные приложения, где важен каждый байт трафика
- Агрегация данных из нескольких источников
- Быстрая эволюция API без версионирования
- Несколько клиентов (web, mobile, desktop) с разными требованиями к данным

### Итоговое заключение

REST и GraphQL — не взаимоисключающие технологии. REST отлично подходит для простых, хорошо кэшируемых API, особенно когда клиенты предсказуемы. GraphQL даёт гибкость клиентам и экономит трафик, но требует более сложной инфраструктуры.

В современных проектах часто используется гибридный подход: публичное API через REST, а для сложных внутренних клиентов — GraphQL.

## Контрольные вопросы

### 1. Что такое over-fetching и under-fetching?

**Over-fetching** — ситуация, когда сервер возвращает больше данных, чем нужно клиенту.
*Пример:* В REST при GET /tasks возвращаются все поля задачи, хотя на экране списка нужны только id и title.

**Under-fetching** — ситуация, когда одного запроса недостаточно для получения всех нужных данных, и клиенту приходится делать несколько запросов.
*Пример:* В REST для получения задачи и её комментариев нужно сделать два запроса: GET /tasks/{id} и GET /tasks/{id}/comments.

### 2. Почему GraphQL может быть опасен без ограничений сложности запросов?

GraphQL позволяет клиентам запрашивать любые данные в любой комбинации. Без ограничений это может привести к:

1. **N+1 проблеме** — вложенные запросы могут вызывать множество обращений к БД
2. **DOS-атакам** — злоумышленник может отправить очень сложный запрос
3. **Чрезмерной нагрузке** — глубоко вложенные запросы могут генерировать огромные объёмы данных
4. **Дорогим вычислениям** — запрос может включать поля с тяжёлыми вычислениями

**Решения:** ограничение глубины, лимиты на количество объектов, анализ стоимости запросов, DataLoader.

### 3. Почему кэширование REST обычно проще?

В REST каждый ресурс имеет уникальный URL (например, /tasks/123). Это позволяет использовать стандартное HTTP кэширование:

- Кэширование по URL в браузере или CDN
- Заголовки Cache-Control для управления кэшем
- ETag для валидации кэша
- Прокси-серверы могут кэшировать ответы

В GraphQL все запросы идут на один endpoint (/query), поэтому:
- HTTP кэширование неэффективно
- Требуется кэширование на уровне данных (Redis, Memcached)
- Сложнее инвалидация кэша

### 4. Как отличаются ошибки REST и GraphQL с точки зрения клиента?

**REST:**
- Разные HTTP статусы для разных ошибок (400, 401, 403, 404, 500)
- Клиент должен обрабатывать каждый статус отдельно
- Тело ответа может содержать детали ошибки
- Просто мониторить по статусам

**GraphQL:**
- Всегда HTTP 200
- Ошибки в поле `errors` ответа
- Единый формат ошибок
- Сложнее мониторинг (нужно парсить тело ответа)
- Частичный успех (часть данных может быть получена, часть — нет)

### 5. В каких случаях вы бы выбрали REST, а в каких GraphQL?

**Выбрал бы REST когда:**
- API в основном публичное и используется разными клиентами
- Важно эффективное кэширование на CDN
- Простые CRUD операции без сложных связей
- Команда уже использует REST и нет времени на переобучение
- Нужна простота мониторинга и отладки

**Выбрал бы GraphQL когда:**
- Мобильное приложение с ограниченным трафиком
- Сложные клиенты с разными потребностями в данных
- Агрегация данных из нескольких микросервисов
- Быстрое прототипирование и частые изменения API
- Нужна строгая типизация и самодокументированное API
